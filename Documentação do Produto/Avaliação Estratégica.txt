AVALIAÇÃO ESTRATÉGICA — Tusab
© 2026 CriAugu — Atualizado: Junho 2026

Análise crítica sob seis perspectivas da cadeia de produto digital.
Cada pilar mapeia o que está sólido, o que preocupa e as ações prioritárias.

─────────────────────────────────────────
STATUS GERAL
─────────────────────────────────────────

O produto está tecnicamente maduro para v1.0.
O gargalo agora não é técnico — é de go-to-market.
Cada semana sem a primeira venda é uma semana de validação desperdiçada.

Categoria de mercado: PKM (Personal Knowledge Management)
Conceito de produto: Mentor Digital

PKM descreve o que o produto faz tecnicamente.
Mentor digital descreve o que o usuário experimenta.
A distinção importa na comunicação: PKM é jargão, mentor é imediato.

─────────────────────────────────────────
1. ARQUITETO DE SOFTWARE
─────────────────────────────────────────

SÓLIDO
· Separação limpa: motor de extração / agente RAG / API
· Frontend modularizado em 15 componentes com responsabilidade única
· BM25 local eliminou dependência de embeddings e APIs externas
· Hooks customizados (useStatus, useAgentStatus) isolam side effects
· services/api.js centraliza todas as chamadas HTTP

PREOCUPA
· App.jsx ~1.300 linhas orquestrando ~40 estados — "God Component"
  Vai virar gargalo de manutenção conforme o produto cresce.
  Próxima refatoração: extrair estado global para Context API ou Zustand.

· api_Tusab.py ~700 linhas sem separação rotas/serviços/repositório
  Para a fase atual é aceitável, mas dificulta testes e crescimento.
  Arquitetura ideal: routers FastAPI separados por domínio.

· Ausência total de testes automatizados
  Qualquer refactor é feito "no escuro". Risco crescente.

AÇÕES PRIORITÁRIAS
○ Implementar testes de integração mínimos nos endpoints críticos
  (extração, indexação, chat) — protege contra regressões
○ Migrar estado global do App.jsx para Context API
  quando o produto entrar em fase de crescimento
○ Separar api_Tusab.py em routers por domínio na v2

DÉBITO TÉCNICO CRÍTICO
Ausência de testes automatizados. Investimento de maior retorno agora.

2. PRODUCT MANAGER
─────────────────────────────────────────

SÓLIDO
· Proposta de valor clara no Day 1: extrai canal, indexa, funciona
· Jornada documentada em 6 passos
· Base de documentos próprios como driver natural de retenção
  (cada arquivo adicionado aumenta custo de troca)

PREOCUPA

  Não sabemos quantos usuários têm, onde travam, qual feature usam.
  O produto evolui por intuição. Posthog precisa sair do backlog.

· Jornada com atritos de ativação
  instalar → extrair → indexar → perguntar ainda gera dúvidas.
  O onboarding explica mas não guia. O usuário ainda se pergunta
  "o que faço agora?" em momentos críticos.

· Day 7 e Day 30 não estão pensados
  O produto capta bem no Day 1. Retenção de médio prazo não está mapeada.
  O que traz o usuário de volta além da primeira extração?

AÇÕES PRIORITÁRIAS

○ Criar onboarding interativo contextual (tooltip por aba, não só modal)
○ Definir métrica de retenção: "usuário que adicionou 2+ fontes diferentes"
○ Medir: tempo até primeiro chat bem-sucedido (Time to Value)

4. ENGENHEIRO DE SEGURANÇA
─────────────────────────────────────────

SÓLIDO
· Dados nunca saem da máquina por design (local-first)
· OAuth com escopo drive.file mínimo
· Sem servidor central para ser atacado
· Extração roda no IP residencial do usuário
· Electron configurado corretamente: contextIsolation=true, nodeIntegration=false
· subprocess usa lista (nunca shell=True) — sem shell injection
· .gitignore protege credentials, token, .env, crash.log
· Sanitização de nome de arquivo com regex já implementada

AUDITORIA COMPLETA — Junho 2026
Realizada análise de código end-to-end. Achados por severidade:

CRÍTICO (bloqueia distribuição)
① CORS aberto — allow_origins=["*"] com allow_credentials=True
  Qualquer site ou processo local pode chamar qualquer endpoint da API.
  Arquivo: api_Tusab.py linha 27
  Fix: restringir a ["http://localhost:8001", "http://127.0.0.1:8001"]

② yt-dlp recebe playlist ID sem validação de formato
  ID vem da saída do próprio yt-dlp mas sem sanitização posterior.
  Um canal malicioso poderia injetar argumentos via ID de playlist.
  Arquivo: motor_Tusab.py linha 503
  Fix: validar com regex ^[A-Za-z0-9_\-]{10,50}$ antes de montar URL

ALTO (corrigir antes do lançamento)
③ Upload sem limite de tamanho
  await arquivo.read() sem MAX_FILE_SIZE → DoS por arquivo gigante.
  Arquivo: api_Tusab.py linha 767
  Fix: limitar a 50 MB, rejeitar com mensagem clara

④ Path traversal no delete de arquivo
  os.path.join() sem validação os.path.realpath() pode sair do subdir.
  Arquivo: api_Tusab.py linha 864
  Fix: verificar que caminho real está dentro do subdiretório permitido

MÉDIO (corrigir antes da v1.0)
⑤ dangerouslySetInnerHTML sem sanitização
  Arquivo: App.jsx linha 726 — string de i18n renderizada como HTML raw
  Fix: remover ou usar DOMPurify

⑥ Chaves de API em plaintext (agent_config.json)
  Para MVP é aceitável. Para v1.0 migrar para keychain do OS via keytar.
  Já mapeado no roadmap de médio prazo.

BAIXO (backlog)
⑦ Campos Pydantic sem limite de tamanho (mensagem, historico)
  Fix: Field(..., max_length=5000) e Field(default=[], max_length=20)

AÇÕES PRIORITÁRIAS

7. LGPD E PRIVACIDADE
─────────────────────────────────────────

Revisão realizada em Junho 2026 com foco na Lei Geral de Proteção de Dados
(Lei 13.709/2018) e nas obrigações relacionadas à coleta, transferência e
armazenamento de dados pessoais.

CONTEXTO
O Tusab é local-first — a maior parte dos dados nunca sai da máquina.
Mas três superfícies de coleta existem:
  1. Posthog Analytics (opt-in) — eventos de uso enviados para servidores fora do BR
  2. APIs externas (Gemini, OpenAI, Anthropic, Groq) — mensagens do chat + contexto da base
  3. Google Drive OAuth — autenticação e sincronização de arquivos

ACHADOS — AUDITORIA LGPD (Junho 2026)

CRÍTICO
· Dado potencialmente pessoal enviado ao Posthog sem declaração
  O evento Analytics.canalConfigurado envia o parâmetro {canal: url}
  para o Posthog. A URL do canal revela qual criador o usuário está
  monitorando — dado de comportamento com contexto profissional.
  Não foi declarado na ConsentModal nem na política de privacidade.
  LGPD Art. 7 exige base legal e ciência do titular para cada dado coletado.
  Fix: remover o parâmetro {canal} do payload do evento.
  Alternativa se necessário para analytics: enviar apenas hash do domínio.

ALTO
· Transferência internacional não informada na UI de configuração de API
  Ao configurar Gemini, OpenAI, Anthropic ou Groq, as mensagens de chat
  e trechos da base de conhecimento saem da máquina para servidores fora
  do Brasil (EUA, UE). LGPD Art. 33 exige informar sobre transferência
  internacional antes da coleta.
  A UI atual não faz nenhum aviso sobre isso — o usuário não tem ciência
  de que seu conteúdo vai sair da máquina ao usar provedores externos.
  Fix: adicionar alerta contextual no card de cada provedor externo.

· Consentimento analytics não revogável pela UI
  A ConsentModal solicita opt-in na primeira execução.
  Não há mecanismo de revogação posterior — violação do Art. 18, VI da LGPD
  ("direito à revogação do consentimento a qualquer momento").
  Fix: adicionar toggle "Desativar analytics" nas configurações do agente.

MÉDIO
· Política de privacidade ausente
  Obrigatória antes do OAuth em produção (requisito do Google para apps
  com acesso ao Drive) e antes de qualquer distribuição pública.
  Precisa cobrir: dados locais, Posthog, APIs externas, Drive OAuth,
  direitos do titular, canais de contato.

· Onboarding não menciona saída de dados ao configurar API externa
  O passo "Configure sua chave de API" não informa ao usuário que,
  ao usar provedores como Gemini ou OpenAI, o conteúdo do chat e da
  base sai da máquina. Informação mínima exigida pelo princípio da
  transparência (Art. 6, VI da LGPD).

BAIXO
· Chave Posthog exposta no bundle JS (web_interface/.env → build)
  Baixo risco: a chave tem somente permissão de escrita (captura eventos,
  não lê dados de outros usuários). Mas expô-la em texto claro num bundle
  distribuído não é ideal. Para v1.0: injetar via variável de ambiente
  no processo Electron sem incluir no bundle estático.

AÇÕES PRIORITÁRIAS
✅ Remover {canal} do payload de Analytics.canalConfigurado — CRÍTICO
✅ Aviso de transferência internacional nos cards de API externa — ALTO
✅ Toggle de revogação de consentimento nas configurações — ALTO
○ Política de privacidade antes do lançamento público — MÉDIO
○ Menção de transferência de dados no onboarding (step de API) — MÉDIO

8. REVISÃO FUNCIONAL DE PRODUTO
─────────────────────────────────────────

Revisão em Junho 2026 focada em paridade entre o que a UI promete
e o que o backend entrega, lacunas de funcionalidade e coerência
entre versão declarada e estado real do produto.

ACHADOS (Junho 2026)

CRÍTICO
· Toggle "Busca Ampla" tem zero efeito funcional
  O estado buscaAmpla é gerenciado corretamente no frontend (useState,
  tooltip, visual feedback) mas o parâmetro busca_ampla não é incluído
  no payload enviado ao backend em sendChatStream (~App.jsx linha 453).
  Todas as perguntas rodam em modo restrito, independente do toggle.
  Impacto: funcionalidade anunciada, zero efeito real. Se o usuário
  notar, quebra a confiança no produto inteiro.
  Fix: adicionar busca_ampla: buscaAmpla no objeto do payload.

ALTO
· Groq disponível no backend mas invisível na UI de provedores
  O backend suporta Groq (free tier, sem cartão), mas a UI lista apenas
  Gemini, OpenAI e Anthropic. Groq é especialmente relevante para usuários
  sem acesso a cartão de crédito internacional (público BR principal).
  Fix: adicionar card Groq na seção de provedores do AgentConfig.

· Versão declarada no footer não condiz com versão real do produto
  Footer exibe "Versão 2.0 Web" em todos os idiomas (pt, en, es).
  O produto está em v0.4.3. A discrepância enfraquece credibilidade
  ao mostrar a ferramenta para clientes ou ao tirar screenshot para
  divulgação.
  Fix: atualizar chave footer.version nos três arquivos de i18n.

MÉDIO
· Analytics.chatPergunta() definido mas nunca invocado
  O método existe em analytics.js e está previsto no plano de telemetria,
  mas nenhuma chamada foi adicionada ao fluxo de envio de mensagem.
  Isso cega a principal métrica de valor (Time to First Value Answer).
  Fix: chamar Analytics.chatPergunta() no handler de envio do chat.

· Nenhum indicador de "base desatualizada" na home card do agente
  Após extrair novos vídeos, o usuário não tem feedback visual na home
  de que precisa reindexar. Só descobre quando entra na aba do agente.
  Fix: badge "X vídeos novos para indexar" na home card do agente.

· Erro de Ollama offline não diferenciado do erro genérico de servidor
  Quando o Ollama trava, o chat exibe "Erro ao conectar com servidor" —
  o mesmo erro de qualquer falha de rede. O usuário não sabe se é o
  Ollama, o FastAPI ou a conexão.
  Fix: detectar status Ollama no payload de erro e exibir mensagem específica.

VALIDAÇÃO PELO ECOSSISTEMA
─────────────────────────────────────────

[Junho 2026] Google lança Gemma 4 12B via Ollama

Relevância estratégica: o maior player em IA está construindo na direção
local-first — mesma filosofia do Tusab. Isso não é coincidência,
é validação de mercado.

Impacto por pilar:

ARQUITETO
· 256K tokens de contexto abre modo "full context" para bases pequenas
  (< 50 arquivos), sem necessidade de seleção BM25
· BM25 permanece essencial para bases grandes (100+ vídeos extraídos)
· Gemma 4 multimodal (imagem + áudio) pode substituir pipelines separados
  de OCR e Whisper, reduzindo débito técnico futuro

PRODUCT MANAGER
· Seletor de modelo Ollama (estava na Fase 3) sobe para próximo sprint
  — usuários com Gemma 4 já instalam pelo próprio fluxo do Ollama,
  mas o Tusab precisa listar e permitir selecionar
· Requisito de hardware aumenta: Gemma 4 12B exige 8GB RAM
  — llama3.2:1b deve continuar como padrão, Gemma 4 como opção avançada

ESTRATEGISTA
· A janela de timing identificada como risco continua se abrindo
  O ecossistema local está amadurecendo rapidamente
  Criadores ainda não têm PKM que aproveite isso — momento de ir ao mercado
· Risco de commoditização: modelos melhores ficam cada vez mais acessíveis
  O diferencial defensável do Tusab não é o modelo — é a camada
  de extração (YouTube nativo), indexação e citação de fontes

CONCLUSÃO
O Tusab não compete com o Gemma 4. São camadas diferentes:
  Gemma 4 → motor de linguagem
  Tusab → infraestrutura de conhecimento pessoal

Quanto melhor o motor, mais valioso fica o Tusab como camada de cima.

O Google está validando a filosofia, não competindo com o produto.
Cada avanço do ecossistema local — Gemma 4, LiteRT-LM, Ollama melhorando —
aumenta o teto do Tusab, não diminui.

Por quê? Porque o usuário final não vai configurar Ollama, baixar o Gemma 4
e montar um pipeline de RAG. Ele vai abrir o Tusab, colar o link do canal
e perguntar. O trabalho técnico invisível é exatamente o que o Tusab resolve.

O NotebookLM faz algo parecido mas deliberadamente não quer o nicho local/privado
— é um produto Google, sempre dependerá dos servidores do Google.
Esse espaço fica aberto. O Tusab ocupa.
