STATUS GERAL

─────────────────────────────────────────
1. ARQUITETO DE SOFTWARE
─────────────────────────────────────────

SÓLIDO
· Separação limpa: motor de extração / agente RAG / API
· Frontend modularizado em 15 componentes com responsabilidade única
· BM25 local eliminou dependência de embeddings e APIs externas
· Hooks customizados (useStatus, useAgentStatus) isolam side effects
· services/api.js centraliza todas as chamadas HTTP

PREOCUPA
· App.jsx ~1.300 linhas orquestrando ~40 estados — "God Component"
  Vai virar gargalo de manutenção conforme o produto cresce.
  Próxima refatoração: extrair estado global para Context API ou Zustand.

· api_brainiac.py ~700 linhas sem separação rotas/serviços/repositório
  Para a fase atual é aceitável, mas dificulta testes e crescimento.
  Arquitetura ideal: routers FastAPI separados por domínio.

· Ausência total de testes automatizados
  Qualquer refactor é feito "no escuro". Risco crescente.

AÇÕES PRIORITÁRIAS
○ Implementar testes de integração mínimos nos endpoints críticos
  (extração, indexação, chat) — protege contra regressões
○ Migrar estado global do App.jsx para Context API
  quando o produto entrar em fase de crescimento
○ Separar api_brainiac.py em routers por domínio na v2

DÉBITO TÉCNICO CRÍTICO
Ausência de testes automatizados. Investimento de maior retorno agora.

─────────────────────────────────────────
2. PRODUCT MANAGER
─────────────────────────────────────────

SÓLIDO
· Proposta de valor clara no Day 1: extrai canal, indexa, funciona
· Jornada documentada em 6 passos
· Base de documentos próprios como driver natural de retenção
  (cada arquivo adicionado aumenta custo de troca)

PREOCUPA
· Métricas inexistentes
  Não sabemos quantos usuários têm, onde travam, qual feature usam.
  O produto evolui por intuição. Posthog precisa sair do backlog.

· Jornada com atritos de ativação
  instalar → extrair → indexar → perguntar ainda gera dúvidas.
  O onboarding explica mas não guia. O usuário ainda se pergunta
  "o que faço agora?" em momentos críticos.

· Day 7 e Day 30 não estão pensados
  O produto capta bem no Day 1. Retenção de médio prazo não está mapeada.
  O que traz o usuário de volta além da primeira extração?

AÇÕES PRIORITÁRIAS
○ Implementar Posthog (opt-in) — saber onde os usuários travam
○ Criar onboarding interativo contextual (tooltip por aba, não só modal)
○ Definir métrica de retenção: "usuário que adicionou 2+ fontes diferentes"
○ Medir: tempo até primeiro chat bem-sucedido (Time to Value)

─────────────────────────────────────────
3. DESIGNER DE PRODUTO (UX)
─────────────────────────────────────────

SÓLIDO
· Home com cards é clara e contextual (badges com dados reais)
· Chat flutuante é o padrão correto — acessível de qualquer aba
· Sidebar recolhível resolve o problema de espaço
· Tema automático via prefers-color-scheme

PREOCUPA
· Indexação desatualizada sem indicação visual
  O usuário não sabe que tem novos vídeos para indexar.
  Falta: badge "X novos vídeos desde a última indexação"
  na aba Configurar Agente ou na home.

· Toggle "Restrita/Ampla" sem contexto para usuário novo
  Está no header do chat sem explicação inline.
  Um usuário novo não entende o que significa sem ler documentação.
  Falta: tooltip ou descrição na primeira vez.

· Estado da base pouco transparente
  Para saber o status da base, o usuário precisa navegar até
  Configurar Agente. Deveria ser visível na home ou no chat.

· Mobile inviável para extração
  Sidebar, log em tempo real e campos de configuração são inutilizáveis
  em tela pequena. Não é prioridade agora (público usa desktop),
  mas precisa estar no radar para expansão.

AÇÕES PRIORITÁRIAS
○ Badge "X novos vídeos para indexar" — indicador de base desatualizada
○ Tooltip no toggle Busca Ampla/Restrita — explica na primeira vez
○ Indicador de estado da base na home card "Configurar Agente"

─────────────────────────────────────────
4. ENGENHEIRO DE SEGURANÇA
─────────────────────────────────────────

SÓLIDO
· Dados nunca saem da máquina por design (local-first)
· OAuth com escopo drive.file mínimo
· Sem servidor central para ser atacado
· Extração roda no IP residencial do usuário

PREOCUPA
· API sem autenticação
  Qualquer processo local pode chamar os endpoints em localhost:8001.
  Aceitável para desktop single-user. Problemático se migrar para web
  ou modo servidor institucional.

· Chaves de API em texto plano (agent_config.json)
  Para uso pessoal é ok. Para distribuição comercial deveria usar
  o keychain do sistema operacional (Electron tem suporte nativo via
  keytar). Evita exposição se o arquivo for copiado inadvertidamente.

· brainiac_crash.log commitado no git
  Pode expor caminhos, erros e detalhes do sistema em repositório.
  Deve ser adicionado ao .gitignore imediatamente.

AÇÕES PRIORITÁRIAS
✅ Adicionar brainiac_crash.log ao .gitignore (ação imediata)
○ Migrar armazenamento de chaves API para keychain do OS via keytar
○ Adicionar token de autenticação local para a API quando houver
  modo servidor institucional (fase 4)

5. ESTRATEGISTA DE NEGÓCIO
─────────────────────────────────────────

SÓLIDO
· Momento certo: onda PKM ainda crescendo, NotebookLM validou mercado
· Obsidian validou que usuários pagam por ferramentas de conhecimento local
· Diferencial real e defensável: YouTube nativo + local + gratuito
· Tração validada: AUVP comprou a ideia antes do produto existir

PREOCUPA
· Sem sistema de licença = sem receita
· Sem telemetria = sem dados para iterar
· Sem landing page = sem pipeline de aquisição
  Os três precisam existir em paralelo, não sequencialmente.

· Risco de timing
  O produto está pronto para v1.0. O mercado está aberto agora.
  Cada semana adicional de desenvolvimento sem venda é risco de
  perder a janela de first-mover advantage no segmento criadores BR.

AÇÕES PRIORITÁRIAS (ordem de impacto)
1. Primeira venda B2B formal (zero código necessário)
   → Demo com criador educacional, contrato simples, nota fiscal
2. Landing page mínima com proposta de valor e formulário de contato
3. Sistema de licença Lemon Squeezy (após primeira venda validar preço)
4. Telemetria Posthog opt-in (após distribuição)

─────────────────────────────────────────
6. ENGENHEIRO DE CONFIABILIDADE (SRE)
─────────────────────────────────────────

SÓLIDO
· Auto-update do yt-dlp via GitHub API no startup
· Migração automática cerebro_txt → cerebro/youtube
· Loading screen com feedback em tempo real durante setup
· Electron gerencia lifecycle do backend e Ollama

PREOCUPA
· Nenhum mecanismo de recovery para agent_index/ corrompido
  Se o índice corromper, o usuário perde o chat sem mensagem clara.
  Falta: detecção de corrupção + botão "Recriar índice" na UI.

· Nenhum backup automático da base de conhecimento
  Se o usuário apagar cerebro/ acidentalmente, perde tudo.
  Um export periódico (.zip da pasta cerebro/) seria diferencial de
  confiança — especialmente para o pitch institucional.

· Sem monitoramento de saúde dos processos
  Se o Ollama travar silenciosamente, o chat simplesmente não responde.
  O usuário não recebe feedback claro de "Ollama offline".

AÇÕES PRIORITÁRIAS
○ Adicionar detecção de índice corrompido + recovery automático
○ Botão "Exportar base" na aba Repositório (zip de cerebro/)
○ Indicador de saúde do Ollama mais visível no painel do agente
○ brainiac_crash.log no .gitignore (compartilhado com pilar de segurança)

VALIDAÇÃO PELO ECOSSISTEMA
─────────────────────────────────────────

[Junho 2026] Google lança Gemma 4 12B via Ollama

Relevância estratégica: o maior player em IA está construindo na direção
local-first — mesma filosofia do BrainIAc. Isso não é coincidência,
é validação de mercado.

Impacto por pilar:

ARQUITETO
· 256K tokens de contexto abre modo "full context" para bases pequenas
  (< 50 arquivos), sem necessidade de seleção BM25
· BM25 permanece essencial para bases grandes (100+ vídeos extraídos)
· Gemma 4 multimodal (imagem + áudio) pode substituir pipelines separados
  de OCR e Whisper, reduzindo débito técnico futuro

PRODUCT MANAGER
· Seletor de modelo Ollama (estava na Fase 3) sobe para próximo sprint
  — usuários com Gemma 4 já instalam pelo próprio fluxo do Ollama,
  mas o BrainIAc precisa listar e permitir selecionar
· Requisito de hardware aumenta: Gemma 4 12B exige 8GB RAM
  — llama3.2:1b deve continuar como padrão, Gemma 4 como opção avançada

ESTRATEGISTA
· A janela de timing identificada como risco continua se abrindo
  O ecossistema local está amadurecendo rapidamente
  Criadores ainda não têm PKM que aproveite isso — momento de ir ao mercado
· Risco de commoditização: modelos melhores ficam cada vez mais acessíveis
  O diferencial defensável do BrainIAc não é o modelo — é a camada
  de extração (YouTube nativo), indexação e citação de fontes

CONCLUSÃO
O BrainIAc não compete com o Gemma 4. São camadas diferentes:
  Gemma 4 → motor de linguagem
  BrainIAc → infraestrutura de conhecimento pessoal

Quanto melhor o motor, mais valioso fica o BrainIAc como camada de cima.
