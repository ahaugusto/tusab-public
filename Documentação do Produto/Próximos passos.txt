CATEGORIA E CONCEITO
─────────────────────────────────────────

Categoria de mercado: PKM (Personal Knowledge Management)
Conceito de produto: Mentor Digital

PKM é como o mercado técnico classifica o segmento.
Concorrentes vivem nessa categoria: Notion, Obsidian, NotebookLM.

Mentor digital é como o usuário experimenta o produto.

A distinção:
  PKM genérico → ferramenta de organização (você ainda precisa procurar)
  Mentor digital → interlocutor ativo (você pergunta, ele responde)

O problema central que resolve: você salva muito, mas não acessa quando
precisa. A busca é ruim. Você não lembra onde está. Acaba relendo tudo.

O Tusab resolve isso: em vez de procurar, você consulta um mentor
que conhece exatamente o conteúdo que você escolheu — e só esse.
Resposta sempre com citação verificável da fonte original.

Nenhum concorrente combina: YouTube em escala + local + custo zero.

ONDE ESTAMOS (v0.4.0)
─────────────────────────────────────────

O Tusab evoluiu de "ferramenta de YouTube" para
sistema de gestão de conhecimento pessoal multi-fonte:

• Base de conhecimento: YouTube + PDF + DOCX + MD + TXT + texto colado
• Repositório com gestão via interface (adicionar, visualizar, deletar)
• Relatório de extração por canal com stats e tabela de vídeos
• Home screen com cards de navegação contextual
• Chat flutuante acessível de qualquer aba
• Provedor Groq (llama-3.1-70b gratuito) como alternativa ao Ollama
• Agente RAG local com BM25 + camada de anti-alucinação
• Extração de qualquer canal YouTube com atribuição ao criador
• 4 abas: Extração · Repositório · Relatório · Configurar Agente

✅ Botão "Exportar base" — descartado
   Redundante: pasta cerebro/ já é local (acesso direto) e o Drive
   já cobre o compartilhamento com outros usuários. Sem valor adicional.

✅ Responsividade e acessibilidade (v2.0.1)
   Layout responsivo: breakpoints md (768px) e lg (1024px) em toda a UI.
   Touch targets ≥ 44px, aria-labels em botões de ícone, tabelas com
   caption sr-only, focus trap em modais, aria-expanded em painéis.

✅ Query expansion para provedores rápidos (v2.1.1)
   BM25 agora usa variações da pergunta geradas pelo LLM (Groq, OpenAI,
   Anthropic, Gemini) para cobrir sinônimos e paráfrases. Desabilitado
   para Ollama (modelos pequenos geram expansões de baixa qualidade e
   adicionam 10–15s de latência). Expansão propaga para busca multi-canal.
   Ver: Documentação do Produto/BM25 e Query Expansion.txt

✅ Suporte a imagens e áudio no repositório (v2.1.0)
   Imagens: Ollama multimodal (llava/gemma3) → fallback Tesseract OCR.
   Áudio: transcrição local com faster-whisper (modelo base, CPU).
   Dependências opcionais: pytesseract (pip) + Tesseract binário; faster-whisper (pip).
   O conteúdo extraído entra no BM25 igual a qualquer outro documento.

✅ Fila de extração sequencial (v2.2.0)
   Usuário enfileira N canais → motor extrai um por um automaticamente.
   Cancelamento limpa a fila; erro num canal não interrompe os seguintes.
   Backend: extraction_queue + queue_lock em AppState; run_motor() em loop while True.
   Endpoints: POST /queue/add, DELETE /queue/clear, GET /queue.
   UI: input + botão + na sidebar, lista numerada, botão limpar fila.

QUANDO O PRODUTO ESTIVER PRONTO PARA LANÇAR
─────────────────────────────────────────

Estes itens ficam para quando a decisão de ir ao mercado for tomada:

A. Publicar OAuth no Google Cloud Console
   → Testing → Production (privacy policy e TOS prontos)
   → Desbloquia Drive para qualquer usuário Google

B. Registro INPI
   → inpi.gov.br > Programa de Computador (~R$90)
   → Titularidade: CriAugu CNPJ 65.131.075/0001-57

C. Gerar primeiro instalador oficial
   → setup-env.ps1 → npm run build web_interface → npm run build electron
   → Publicar no GitHub Releases para auto-update

D. Landing page mínima
   → Proposta de valor clara, vídeo demo, formulário de contato
   → Gumroad ou Lemon Squeezy como checkout
   → Sem essa página, não há pipeline de aquisição

E. Primeira venda B2B formal
   → Demo com criador educacional (50k–500k inscritos)
   → Contrato simples, nota fiscal, feedback documentado
   Nota: a AUVP é uma proposta separada (chatbot embarcado na plataforma
   deles) — não é canal de venda do Tusab. Prospecção própria necessária.

PRÓXIMO SPRINT — PRODUTO
─────────────────────────────────────────

#1  Busca web no chat (v2)
    Dois modos além do Restrito atual:
    · Amplo (Opção A — implementar logo): base como referência + LLM complementa
      com conhecimento de treinamento. Sem dependência externa, funciona offline.
    · Web (Opção B — médio prazo): integração com Brave Search API
      → 2.000 buscas/mês gratuitas, API key do usuário (console.api.search.brave.com)
      → DuckDuckGo como fallback sem key (parser HTML, frágil)
      → Resultado: snippets da web + chunks BM25 no mesmo contexto
      → Interface: campo para key Brave nas configurações do agente

✅ Modelos Ollama configuráveis pelo usuário (v0.4.3)
    Painel "Configurações Avançadas" colapsável em Configurar Agente.
    Lista modelos instalados, seleção persiste no config.
    Para Gemma 4: ollama pull gemma3:12b → selecionar no dropdown.

#3  Imagens na base
    → Caminho A (curto prazo): Gemma 4 12B via Ollama já processa imagens
      nativamente — sem pipeline separado, basta o usuário ter o modelo
    → Caminho B (fallback): OCR com Tesseract para usuários sem GPU
    → Abre: fotos de slides, capturas de tela, infográficos

#4  Sistema de licença (Lemon Squeezy)
    → Tela de ativação no Electron
    → Validação HTTP simples
    → Pré-requisito: ter tração com vendas antes de implementar

#5  Servidor MCP
    → Expõe a base RAG como ferramenta para agentes externos
    → Usuário conecta Tusab ao Claude Code, Cursor, etc.
    → O Tusab vira uma "ferramenta de busca" para qualquer agente

─────────────────────────────────────────
ECOSSISTEMA LOCAL — VALIDAÇÃO EXTERNA
─────────────────────────────────────────

Em Junho 2026, o Google lançou o Gemma 4 12B:
· Modelo multimodal (texto + imagem + áudio)
· 256K tokens de contexto
· Roda em 8GB de RAM via Ollama
· Apache-2.0 — livre para uso comercial

Convergências diretas com o Tusab:

1. Valida a aposta local-first
   O maior player em IA está construindo na mesma direção.
   Privacidade + autonomia offline está se tornando padrão de mercado,
   não nicho técnico.

2. Roda via Ollama — nosso provedor padrão
   Usuários com Gemma 4 instalado já usam via Tusab sem alteração.
   O seletor de modelos (item #2 do próximo sprint) libera isso.

3. 256K de contexto muda a equação do RAG
   O BM25 existe porque contextos de 4K–8K exigiam seleção de chunks.
   Com 256K, bases pequenas-médias cabem inteiras no contexto.
   Oportunidade: modo "full context" para usuários com base < 50 arquivos.
   O BM25 continua relevante para bases grandes (100+ vídeos extraídos).

4. Multimodal simplifica o roadmap
   Imagens e áudio (Fases 2–3) podem usar Gemma 4 diretamente,
   sem construir pipelines separados de OCR ou ASR.

O que não muda: o Tusab não é o modelo.
É a infraestrutura de conhecimento. Gemma 4 é um motor melhor
que podemos usar. Nossa proposta de valor — extrair YouTube,
indexar documentos, citar fontes — é o que o modelo por si só não faz.

PRINCÍPIO QUE NÃO MUDA
─────────────────────────────────────────

A extração via yt-dlp tem que continuar local.
A execução distribuída na máquina de cada usuário é a proteção natural
contra bloqueios do YouTube — e o diferencial competitivo real do Tusab.

Tudo mais pode evoluir. Esse princípio não.
